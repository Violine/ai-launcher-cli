## 1. Введение

### 1.1. Назначение документа

Настоящий документ является техническим заданием на разработку консольной утилиты AI Launcher CLI, предназначенной для централизованного управления AI-инструментами разработки в корпоративной среде.

### 1.2. Цели проекта

- Унификация доступа к различным AI-ассистентам (Claude Code, OpenCode, Aider и др.)
- Централизованное управление API-ключами и конфигурацией
- Сбор телеметрии для анализа использования AI-инструментов
- Автоматическое обновление утилиты без участия пользователя
- Снижение порога входа для новых сотрудников

### 1.3. Целевая аудитория

Разработчики компании, использующие AI-ассистенты в повседневной работе. Предполагается техническая грамотность пользователей и базовое знание командной строки.

### 1.4. Термины и определения

| Термин | Определение |
|--------|-------------|
| TUI | Text User Interface — текстовый пользовательский интерфейс |
| CLI | Command Line Interface — интерфейс командной строки |
| OTEL | OpenTelemetry — стандарт для сбора телеметрии |
| LLM | Large Language Model — большая языковая модель |
| ENV | Environment Variables — переменные окружения |

## 2. Общее описание системы

### 2.1. Архитектура

AI Launcher представляет собой standalone CLI-приложение, написанное на Go, с TUI-интерфейсом в стиле MS-DOS/Norton Commander. Приложение распространяется как единый бинарный файл без внешних зависимостей.

#### 2.1.1. Компоненты системы

- **TUI Module** — текстовый интерфейс на базе Bubble Tea
- **Config Manager** — управление конфигурацией (YAML)
- **Tool Registry** — реестр доступных AI-инструментов
- **Process Executor** — запуск и управление процессами
- **Telemetry Collector** — сбор и отправка метрик (OTEL)
- **Auto-Updater** — модуль автообновления
- **API Client** — взаимодействие с LLM Proxy

#### 2.1.2. Диаграмма компонентов

```
┌─────────────────────────────────────────────────────────┐
│                     AI LAUNCHER CLI                     │
├─────────────┬─────────────┬─────────────┬───────────────┤
│  TUI Module │ Config Mgr  │ Tool Registry│ Auto-Updater │
├─────────────┴─────────────┴─────────────┴───────────────┤
│              Process Executor + ENV Manager             │
├─────────────────────────────────────────────────────────┤
│         Telemetry Collector (OpenTelemetry)             │
└─────────────────────────────────────────────────────────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │ LLM Proxy│    │ OTEL     │    │ Update   │
    │ API      │    │ Collector│    │ Server   │
    └──────────┘    └──────────┘    └──────────┘
```

### 2.2. Технологический стек

| Компонент | Технология |
|-----------|------------|
| Язык | Go 1.21+ |
| TUI Framework | Bubble Tea + Lip Gloss |
| Конфигурация | YAML (gopkg.in/yaml.v3) |
| Телеметрия | OpenTelemetry SDK for Go |
| Автообновление | go-github + semver |

## 3. Функциональные требования

### 3.1. Управление API-ключами

#### 3.1.1. Ввод и валидация ключа

- **FR-101:** При первом запуске система должна запросить API-ключ
- **FR-102:** Ввод ключа должен быть замаскирован символами `*`
- **FR-103:** Система должна валидировать формат ключа перед сохранением
- **FR-104:** При невалидном ключе — отображение ошибки и повторный ввод

#### 3.1.2. Хранение ключа

- **FR-105:** Ключ сохраняется в `~/.config/ai-launcher/config.yaml`
- **FR-106:** Файл конфигурации имеет права доступа 0600
- **FR-107:** Возможность сброса ключа через интерфейс (клавиша F7)

### 3.2. Получение списка моделей

- **FR-201:** После ввода ключа — запрос к API для получения доступных моделей
- **FR-202:** Отображение индикатора загрузки с анимацией
- **FR-203:** Динамическое формирование списка инструментов из моделей
- **FR-204:** Кэширование списка моделей на 1 час
- **FR-205:** Возможность принудительного обновления списка (клавиша R)

### 3.3. Список инструментов

#### 3.3.1. Отображение

- **FR-301:** Табличное отображение с колонками: #, Название, Модель, Статус, Избранное
- **FR-302:** Подсветка выбранной строки инвертированными цветами
- **FR-303:** Прокрутка списка клавишами ↑/↓, PgUp/PgDn, Home/End
- **FR-304:** Отображение индикатора позиции (Item X of Y)

#### 3.3.2. Раскрытие строки

- **FR-305:** По нажатию Enter строка раскрывается и показывает кнопки действий
- **FR-306:** Кнопки: [▶ Run] [✎ Edit] [★ Fav] [◉ Toggle]
- **FR-307:** Навигация по кнопкам клавишами ←/→
- **FR-308:** Закрытие раскрытой строки по Esc или переходу на другую строку

### 3.4. Редактирование инструмента

- **FR-401:** Отдельный экран редактирования с полями: Command, Model, ENV-переменные
- **FR-402:** Навигация между полями по Tab/Shift+Tab
- **FR-403:** Чекбоксы Enabled и Favorite
- **FR-404:** Кнопки [Save] и [Cancel]
- **FR-405:** Быстрое сохранение по F2
- **FR-406:** Добавление произвольных ENV-переменных

### 3.5. Запуск инструмента

- **FR-501:** Запуск выбранного инструмента как subprocess
- **FR-502:** Передача сконфигурированных ENV-переменных
- **FR-503:** Автоматическая передача API-ключа в ANTHROPIC_API_KEY
- **FR-504:** Передача stdin/stdout/stderr от launcher к инструменту
- **FR-505:** Возврат в launcher после завершения работы инструмента

### 3.6. Автообновление

- **FR-601:** Проверка обновлений при запуске (в фоне)
- **FR-602:** Сравнение версий по semver
- **FR-603:** Уведомление о наличии новой версии
- **FR-604:** Скачивание и установка с подтверждением пользователя
- **FR-605:** Резервное копирование текущей версии перед обновлением
- **FR-606:** Откат при ошибке обновления
- **FR-607:** Принудительная проверка обновлений через флаг `--check-update`

### 3.7. Обновление MCP для установленных AI-инструментов (приоритет 2)

- **FR-701-MCP:** Получение списка MCP-пакетов из GitLab Package Registry (npm)
- **FR-702-MCP:** Установка и обновление npm-пакетов MCP из registry
- **FR-703-MCP:** Конфигурация registry URL и токена (в config или ENV)
- **FR-704-MCP:** Обновление MCP-конфигураций для AI-агентов: после установки/обновления MCP-пакетов launcher актуализирует конфиги установленных инструментов (Claude, Cursor, OpenCode и др.), чтобы агенты использовали актуальный набор MCP-серверов

*Приоритет реализации: сначала автообновление самого ai-launcher-cli (раздел 3.6), затем функционал обновления MCP.*

### 3.8. Телеметрия (OpenTelemetry)

- **FR-701:** Сбор метрик использования: какие инструменты запускаются, как часто
- **FR-702:** Сбор метрик производительности: время запуска, время работы сессии
- **FR-703:** Трейсинг ключевых операций
- **FR-704:** Отправка данных на корпоративный OTEL Collector
- **FR-705:** Возможность отключения телеметрии через конфиг или ENV
- **FR-706:** Не собирать персональные данные и содержимое запросов к LLM

#### 3.8.1. Собираемые метрики

| Метрика | Тип | Описание |
|---------|-----|----------|
| launcher.start | Counter | Количество запусков launcher |
| tool.launch | Counter | Запуски инструментов (по tool_id) |
| tool.session_duration | Histogram | Длительность сессий (сек) |
| launcher.update_check | Counter | Проверки обновлений |
| launcher.update_install | Counter | Установленные обновления |
| api.models_fetch | Counter | Запросы списка моделей |
| api.models_fetch_latency | Histogram | Латентность API (мс) |

## 4. Нефункциональные требования

### 4.1. Производительность

- **NFR-101:** Холодный старт приложения < 100мс
- **NFR-102:** Отклик UI на действия пользователя < 16мс (60fps)
- **NFR-103:** Размер бинарного файла < 15MB
- **NFR-104:** Потребление RAM в idle < 30MB

### 4.2. Совместимость

- **NFR-201:** Поддержка Linux (Ubuntu 20.04+, RHEL 8+)
- **NFR-202:** Поддержка macOS 12+ (Intel и Apple Silicon)
- **NFR-203:** Поддержка Windows 10+ (WSL и native)
- **NFR-204:** Корректная работа в терминалах: iTerm2, Terminal.app, GNOME Terminal, Windows Terminal

### 4.3. Безопасность

- **NFR-301:** API-ключи хранятся с правами доступа 0600
- **NFR-302:** Ключи не отображаются в логах и телеметрии
- **NFR-303:** Проверка подписи при скачивании обновлений
- **NFR-304:** TLS 1.2+ для всех сетевых соединений

### 4.4. Удобство использования

- **NFR-401:** Интуитивно понятный TUI без необходимости чтения документации
- **NFR-402:** Панель F-клавиш с подсказками на каждом экране
- **NFR-403:** Поддержка vim-клавиш (j/k/h/l) для навигации
- **NFR-404:** Время от установки до первого запуска инструмента < 2 минут

## 6. Внешние интеграции

### 6.1. LLM Proxy API

Взаимодействие с корпоративным LLM Proxy для получения списка доступных моделей.

- **Endpoint:** GET /v1/models  
- **Headers:** Authorization: Bearer {api_key}  
- **Response:** JSON со списком моделей  

### 6.2. OpenTelemetry Collector

Отправка метрик и трейсов на корпоративный OTEL Collector.

- **Protocol:** OTLP/gRPC  
- **Endpoint:** настраивается через OTEL_EXPORTER_OTLP_ENDPOINT  
- **Batching:** каждые 30 секунд или при накоплении 100 событий  

### 6.3. Update Server

Сервер для хостинга обновлений (GitLab Releases или Nexus).

- **Проверка версии:** GET /releases/latest  
- **Скачивание:** GET /releases/download/{version}/{platform}  
- **Checksum:** SHA256 для верификации

## 7. План разработки

### 7.1. Этапы

**Приоритеты:** (1) сначала автообновление ai-launcher-cli, (2) затем обновление MCP для установленных AI-инструментов и конфигов агентов.

| Этап | Содержание | Срок |
|------|------------|------|
| 1 | MVP: TUI + Config + базовый запуск | 2 недели |
| 2 | Интеграция с LLM Proxy API | 1 неделя |
| 3 | OpenTelemetry телеметрия | 1 неделя |
| 4 | **Автообновление ai-launcher-cli** (приоритет 1) | 1 неделя |
| 5 | **Обновление MCP для AI-инструментов и MCP-конфигураций агентов** (приоритет 2) | 1 неделя |
| 6 | Тестирование и документация | 1 неделя |
| 7 | Пилот на группе пользователей | 1 неделя |

Общая оценка: 7 недель до production-ready релиза.

## 8. Критерии приёмки

### 8.1. Функциональные критерии

1. Пользователь может ввести API-ключ и запустить любой доступный инструмент  
2. Список инструментов динамически формируется из API  
3. Редактирование ENV-переменных сохраняется между сессиями  
4. Автообновление корректно работает на всех целевых платформах  
5. Телеметрия успешно отправляется на OTEL Collector  

### 8.2. Нефункциональные критерии

6. Время холодного старта < 100мс на референсной машине  
7. Размер бинарника < 15MB  
8. Успешные end-to-end тесты на Linux, macOS, Windows  
9. Покрытие unit-тестами ≥ 70%
