# Декомпозиция: автообновление (3.6)

Цель: реализовать FR-603–FR-607 так, чтобы каждый шаг можно было проверить тестами и вручную.

**Целевой источник обновлений в проде:** GitLab (Releases/API).  
**Для тестов и отладки:** GitHub Releases — создание тега, релиза и загрузка бинарников по шагам описаны в [RELEASE-AND-TEST-UPDATE.md](./RELEASE-AND-TEST-UPDATE.md).

---

## Текущее состояние

- **FR-601** (проверка в фоне): есть `CheckInBackground(currentVersion, onNewVersion)`; при запуске main дергает callback в stderr.
- **FR-602** (semver): есть `NewerThan(current, available)`, `LatestRelease(repo)` через GitHub API.
- **FR-604–FR-606**: заглушка `Install(downloadURL, currentBinaryPath)` — возвращает "not implemented".
- **FR-607** (`--check-update`): нет.
- Источник релизов: сейчас только GitHub; в ТЗ — GitLab Releases или Nexus (опционально позже).

---

## Декомпозиция по задачам

### 1. Тестируемая проверка обновлений (подложка для FR-603, FR-607)

**Цель:** вынести получение «последней версии» и «URL бинарника» за абстракцию, чтобы в тестах подменять на мок.

**Шаги:**

- 1.1 Ввести в `internal/updater` интерфейс источника обновлений, например:
  - `LatestVersion(repo string) (version string, err error)`
  - `DownloadURL(repo, version string) string`
- 1.2 Текущую логику (GitHub API + `DownloadURL`) оформить как реализацию этого интерфейса (например, `GitHubSource`).
- 1.3 **Репозиторий для обновлений — из настроек:** в `Config` добавить поле `UpdateRepo string` (YAML: `update_repo`). Опционально — переменная окружения `AI_LAUNCHER_UPDATE_REPO`. Если пусто — использовать `DefaultRepo` в коде. Так можно тестировать против своего GitHub без правки кода.
- 1.4 `CheckInBackground` и будущий `--check-update` вызывают не GitHub напрямую, а переданный источник (в production — GitHub или GitLab), а repo берётся из конфига/env или дефолта.

**Тестирование:**

- Unit-тесты: мок возвращает `version = "1.2.3"`, текущая `0.0.0` → `NewerThan` true, callback вызывается; мок возвращает `0.0.0` → callback не вызывается.
- `NewerThan` уже чистая функция — тесты на граничные случаи (без `v`, пререлизы и т.д.) при необходимости.

### 2. FR-607: флаг `--check-update`

**Цель:** явная проверка обновлений без запуска TUI/плагинов.

**Шаги:**

- 2.1 В `cmd/ai-launcher` при разборе аргументов обработать `--check-update` (и при необходимости короткий вариант `-c`).
- 2.2 При наличии флага: вызвать логику проверки (тот же источник, что и для фона), вывести в stdout понятное сообщение (например, «Current: X.Y.Z, Latest: A.B.C» или «No update available») и выйти с кодом 0; при ошибке сети/парсинга — stderr и ненулевой код.

**Тестирование:**

- Запуск бинарника с `--check-update`: проверка вывода и кода выхода (скрипт или go test с `exec.Command`).
- С мок-источником (через переменную окружения или тестовый билд с подменой источника) — предсказуемый вывод.

### 3. FR-603: уведомление о новой версии

**Цель:** пользователь узнаёт о доступном обновлении.

**Варианты:**

- **A (текущий):** оставить вывод в stderr при старте + подсказку «Run ai-launcher autoupdate to install» — уже есть.
- **B:** при запуске TUI показывать баннер/тост «Available update: X.Y.Z» (требует передачи факта «есть обновление» в модель TUI и отрисовки).
- **C:** при выборе пункта «AI Launcher Update» в меню: если обновление есть — сразу экран «Download? [Y/n]»; если нет — сообщение «You are up to date».

**Тестирование:**

- A: уже проверяется вручную.
- B/C: unit-тест модели (например, при `AvailableVersion != ""` в view есть строка с версией); интеграционно — запуск TUI с замоканным «есть обновление».

Рекомендация: сначала зафиксировать A; B или C — отдельной задачей после FR-604.

### 4. Скачивание и установка (FR-604, FR-605, FR-606)

**Цель:** по URL скачать бинарь, сделать backup текущего, заменить бинарь; при ошибке — откат.

**Декомпозиция:**

- 4.1 **Скачивание в temp-файл**
  - Функция `DownloadToFile(url, destPath string) error` (или принять `io.Writer` для тестов).
  - В тестах: `httptest.Server` раздаёт файл; проверяем, что содержимое записанного файла совпадает с ожидаемым.

- 4.2 **Опционально: проверка checksum (NFR-303)**
  - Если сервер отдаёт `*.sha256` — скачать и сравнить с SHA256 скачанного бинарника; при несовпадении — не переходить к замене, удалить temp.

- 4.3 **Backup текущего бинарника (FR-605)**
  - Путь текущего: `os.Executable()` (с учётом симлинков при необходимости).
  - Backup: копия в тот же каталог, например `ai-launcher.bak` или `ai-launcher.{version}.bak`.
  - Тест: создать temp-файл «текущий бинарь», вызвать функцию backup → проверить существование и содержимое `.bak`.

- 4.4 **Замена бинарника (FR-604)**
  - После успешного скачивания (и проверки суммы при наличии): переименовать текущий в `.bak`, перенести скачанный из temp в путь текущего, выставить права исполнения (Unix).
  - На Windows: возможны ограничения перезаписи запущенного exe — рассмотреть запись в temp рядом с exe и подсказку «перезапустите» или использование move при следующем запуске (упрощённый вариант — замена при следующем запуске через wrapper).

- 4.5 **Откат при ошибке (FR-606)**
  - Если после переименования в .bak что-то пошло не так (например, не удалось записать новый файл): восстановить из .bak в исходный путь, удалить temp.
  - Тест: симулировать ошибку после переименования в .bak (например, запись в недоступную директорию) → проверить, что исходный файл восстановлен из backup и содержимое совпадает.

**Интерфейс для тестов:**

- Принимать «путь к текущему бинарнику» и «путь к temp-директории» явно (в production — `os.Executable()` и `os.TempDir()`), чтобы в тестах использовать `t.TempDir()` и не трогать реальный файл.

**Тестирование:**

- Unit: мок HTTP (httptest.Server), мок fs через temp dir — полный сценарий download → backup → replace; сценарий с «ошибка после backup» → rollback.
- Интеграционный (опционально): реальный бинарь в temp, реальный httptest.Server с бинарём — полный цикл Install; проверить, что запускается новый бинарь.

### 5. Подтверждение пользователя (FR-604)

**Цель:** не качать/не ставить без явного согласия.

**Варианты:**

- **CLI:** при `ai-launcher autoupdate` если есть новая версия — вывести «Update X.Y.Z available. Install? [y/N]»; по `y` — вызвать Install.
- **TUI:** при выборе «AI Launcher Update» показать экран с версией и кнопками [Install] [Cancel]; по Install — вызвать Install (и при успехе можно выйти с сообщением «Restart the app»).

**Тестирование:**

- CLI: автоматический тест с поданным на stdin «y» или «n» — проверка, что при «n» Install не вызывается, при «y» — вызывается (с моком Install).
- TUI: unit-тест модели (кнопка Install отправляет команду на установку; кнопка Cancel возвращает в меню).

---

### 6. Сборка и передача версии в бинарь

**Цель:** чтобы `autoupdate.Version` не был всегда `0.0.0`, а соответствовал тегу релиза.

**Шаги:**

- 6.1 В `Makefile` (или скрипте сборки) при `make build-all` передавать `-ldflags "-X github.com/ai-launcher/cli/internal/modules/autoupdate.Version=$(VERSION)"` с `VERSION` из тега или переменной окружения.
- 6.2 В CI/CD при создании релиза собирать артефакты с версией из тега и выкладывать в GitHub Releases (имена файлов уже заложены в `DownloadURL`).

**Тестирование:**

- Собрать с `VERSION=1.0.0`, запустить `--check-update` — в выводе должна быть текущая версия 1.0.0.

## Резюме по тестированию

- **Проверка версий и callback:** мок источника версий + unit-тесты `CheckInBackground` и `NewerThan`.
- **Скачивание:** `httptest.Server` + запись в файл; проверка содержимого.
- **Backup/Replace/Rollback:** все операции в `t.TempDir()`, путь к «текущему бинарнику» и к temp передаются в Install; тесты без реального запущенного exe.
- **Флаг `--check-update`:** `exec.Command` или отдельный тест-бинарник с подменой источника.
- **Подтверждение:** передача вводов по stdin и проверка вызова/невызова Install.

После выполнения шагов 1–7 и 10 автообновление можно полноценно тестировать автоматически и проверять вручную на реальном GitHub Release.
